# docu-export — SPECTRA Constitution
Project: docu-export (Export Documentation System)
Company: Texas Worldwide Oil Services LLC (TXWOS)
SPECTRA level: 3 | Track: bmad_method | Risk score: 7
Last updated: 2026-03-06

---

## 1. PROJECT MISSION

Automate generation of international export compliance documents
(COO, Packing List, Export Invoice, COA, SED) from uploaded Bills of Lading.
Eliminate manual data entry. Ensure documents are accurate, consistent,
and professionally formatted for real clients and real shipments.

---

## 2. HARD CONSTRAINTS (never violate)

- **Production data is live.** MongoDB Atlas contains real shipments for
  real clients. No destructive schema migrations without explicit backup step.
- **Admin-only access.** Only these four emails may authenticate:
  de@txwos.com, talvarez@txwos.com, txwos.tomas@gmail.com, txwos.diego@gmail.com
  No open registration. No weakening of auth guards.
- **No OFAC concerns.** Venezuela shipments are permitted. No sanctions
  screening required. Do not add compliance gates that block normal operations.
- **Notary section is always required in COO.** Notary seal + signature assets
  must be loaded from MongoDB Asset model. If assets are missing, fail with a
  clear error — do not silently omit the notary section.
- **Document numbers are business-critical.** BOL numbers, invoice numbers,
  certificate numbers must be preserved exactly as extracted. No reformatting,
  no truncation, no normalization.
- **All code in /project.** Source lives in ~/projects/dev/docu-export/project.
  The docu-export root is config/docs only.
- **TypeScript everywhere.** No plain JS in src/. Use Zod for all validation.
- **Footer:** "By Ascend Consult LLC all rights reserved" on all pages.
  TXWOS address, WhatsApp, LinkedIn in footer.
- **Header:** TXWOS logo from /public/txwos-logo.png on all document pages.

## 3. ARCHITECTURE DECISIONS (locked)

- **MongoDB is primary storage.** All document data, files (GridFS),
  assets, clients, admin users. Firebase Firestore role to be evaluated —
  until evaluated, do not add new Firestore writes.
- **Firebase Functions handle BOL processing.** Claude vision call lives
  in Firebase Function (processBolDocument). Do not move it to Next.js API routes.
- **pdf-lib for server-side PDF generation.** COO already uses pdf-lib (1756 lines).
  All new document generators must use pdf-lib for consistency.
  Do NOT use @react-pdf/renderer for server-side generation.
- **TanStack Query for client state.** SWR hooks are legacy — do not add new SWR
  usage. Migrate to TanStack Query opportunistically.
- **NextAuth for session management.** Firebase Auth handles SSO layer.
  Both are required. Do not remove either.

## 7. DATA INTEGRITY RULES

- Never delete a Document record — use soft delete or status flags
- GridFS files are immutable — generate new file on regeneration, do not overwrite
- BOL numbers are unique keys — enforce at DB level and application level
- Client RIF numbers are unique per client — validate on create/update
- All monetary values stored as strings with explicit currency — never as floats
- Weight/volume conversions use product-specific densities from context.md
  (PRIMA 600N: 0.8725 kg/L, BS150 GII: 0.8892 kg/L)
